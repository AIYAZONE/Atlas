# 架构演示提纲（30分钟｜混合听众：管理层 + 技术）

## 演示目标
- 管理层：听懂“为什么这样拆、风险如何控制、能否规模化”。
- 技术：听懂“边界在哪里、关键机制是什么、落地顺序怎么走”。

## 讲述主线（3句话开场）
1) Atlas 是内部系统，但产出面向消费者的外部静态站点。  
2) 采用“Java 单入口 + Node 仅渲染/构建”实现治理与交付效率兼得。  
3) 发布体系（版本化产物 + 原子切换 + 回滚）支撑 99.9 与规模化。

### 1. 架构全景（What）（7分钟）
- 讲（管理层能听懂的版本）：  
  - 谁对公网：只有 Java API（统一鉴权/权限/审计/限流）。  
  - 谁在内网：Node Preview/Publisher（渲染、构建、发布执行）。  
  - 站点交付：纯静态 + CDN，流量与后端解耦。  
- 看：

### 2. 为什么是 Java + Node（Why）（4分钟）
- 讲：  
  - Java 单入口更适合做治理（SSO、RBAC、审计、合规、对外契约稳定）。  
  - Node 更适合做渲染/构建（TS 生态、插件与模板能力、与前端协作成本更低）。  
  - 结果：安全边界清晰、职责单一、可独立扩容、可演进。  
- 看：  
  - 技术选型决策（只讲结论与权衡）：[06_ADR_Java_vs_Node.md](../02_Technical_Architecture/06_ADR/06_ADR_Java_vs_Node.md)

### 3. 发布体系与SLA（How，核心卖点）（8分钟）
- 讲：  
  - 发布永远异步：创建 job 立即返回 jobId，不占用 API 长连接。  
  - 版本化产物不可变：写到 releases/version/，上线用指针原子切换。  
  - 回滚分钟级：切回旧指针即可，不需要逐页失效 1000/10000 个 URL。  
  - 并发与限流：Worker 并发受控 + Shopify 429 退避重试，避免雪崩。  
- 看：  
  - 主讲稿：[10_SLA与发布体系（99.9_10000页）.md](../02_Technical_Architecture/10_SLA与发布体系（99.9_10000页）.md)

  - 全局组件批量发布（答疑备用）：[11_全局组件批量发布方案.md](../03_DevOps_Risk/11_Publishing/11_全局组件批量发布方案.md)

### 4. 数据模型真源与可扩展性（5分钟）
- 讲：  
  - Schema First：先定义 Schema，编辑器自动生成 UI，减少硬编码与耦合。  
  - Shopify Compatible：迁移成本低、概念复用、可双向互通。  
  - 扩展性：插件机制与多语言规范支撑长期演进。  
- 看：

### 5. 风险与治理（3分钟）
- 讲：  
  - 安全边界：单入口 + 内网 Node + 最小权限 + Webhook 签名与幂等。  
  - 运维：监控告警 + 审计可追溯。  
  - 外部站点：SEO 与一致性优先，静态交付避免运行时抖动。  
- 看：  
  - 风险清单（挑 2-3 条讲）：[10_核心风险与应对策略.md](../03_DevOps_Risk/10_核心风险与应对策略.md)

### 6. Q&A 缓冲（0-3分钟）
- 预备高频问题与答法（每题 20-40 秒）：  
  - 为什么不用全 Node / 全 Java？（指向 ADR，强调治理 vs 渲染职责分离）  
  - 改 Header/Footer 是否每次都要全站重建？（默认全站 job，必要时评估边缘重写；仍保留批量兜底）  
  - 99.9 怎么保证？（异步 job、原子切换、分钟级回滚、可观测与告警）  
  - 10000 页发布多久？（增量为默认；全量只做灾备；并发与限流参数在 SLA 文档）
