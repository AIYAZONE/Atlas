# Jackery Atlas 产品路线图

> 本文档根据最新的 8 周 MVP 稳健计划更新，旨在明确各阶段的交付目标与验收标准。

## 愿景与定位

* **定位**：面向 Jackery 内部的可视化建站与静态发布基础设施。

* **核心价值**：

  * **降本增效**：替代昂贵 CMS 与插件，提升运营自助能力。

  * **资产沉淀**：将页面数据（JSON Schema）与前端组件（Vue 3）转化为公司自有数字资产。

  * **技术现代化**：基于 Vue 3 + Nuxt 3 栈，消除 Liquid 技术债，提升团队士气与招聘吸引力。

## 阶段规划

### Phase 1：MVP 奠基（Now – 8 Weeks）

> 目标：交付单站点闭环，验证核心技术链路与业务价值。
> 详细计划：[00\_MVP里程碑计划\_详细版.md](./00_MVP里程碑计划_详细版.md)

* **核心交付**：

  * **Builder**：基于 Schema 的可视化编辑器（拖拽/配置/预览/发布）。

  * **Runtime**：Nuxt 3 SSG 静态站点运行时，Lighthouse ≥ 95。

  * **Shopify 适配**：打通 Storefront API 读取商品，Checkout 跳转原生结账页。

  * **发布治理**：CI/CD 流水线、CDN 部署、发布回滚与监控。

* **验收标准**：

  * 非技术人员可自助配置首页并发布。

  * 静态页面可被搜索引擎抓取，结构化数据合规。

  * 购物车闭环完整，数据归属 Shopify 后端。

### Phase 2：能力扩展（3 – 6 Months）

> 目标：完善内容生态与营销能力，支持复杂业务场景。

* **核心交付**：

  * **多页面支持**：集合页（Collection）、博客（Blog）、自定义落地页（Landing Page）。

  * **营销增强**：优惠券逻辑适配、倒计时/弹窗等营销组件库扩充。

  * **性能进阶**：深度优化增量构建（ISR），将发布延迟控制在分钟级。

  * **多语言基础**：路由结构支持（i18n）、词条管理方案。

  * **架构升级**：按需引入 NestJS 中间层处理复杂聚合逻辑（BFF）。

### Phase 3：平台化与生态（6 – 12 Months）

> 目标：实现多站点复用与规模化效益，构建内部开发者生态。

* **核心交付**：

  * **主题迁移工具**：辅助将现有 Liquid 主题转化为 Atlas 组件/Schema。

  * **插件市场**：内部插件标准化（Plugin API），支持第三方功能接入。

  * **A/B 测试**：集成实验平台，支持页面级与组件级变体测试。

  * **权限与审计**：RBAC 细粒度权限控制，操作日志全链路审计。

## 核心 KPI 与验收指标

* **性能体验**

  * Lighthouse（Mobile）：≥ 95

  * 静态首屏时间（FCP）：≤ 1.0s

  * 累积布局偏移（CLS）：≤ 0.1

* **构建与运维**

  * 单页增量构建时间：≤ 5s

  * 全站构建成功率：≥ 99.9%

  * 故障恢复时间（RTO）：≤ 10min（回滚耗时）

* **业务价值**

  * 运营自助率：首页/活动页内容修改 100% 自助

  * 资产沉淀：核心组件库覆盖率 ≥ 80%

## 风险与应对

* **重造轮子成本**：部分 Shopify 插件前端需重写。

  * *应对*：预留 30% 额外工时，MVP 仅实现核心功能，非核心寻找 API 替代方案。

* **发布延迟**：SSG 构建需时间，不如原生 Shopify 即时。

  * *应对*：落地增量构建（ISR），建立发布窗口预期管理。

* **运维复杂度**：需维护数据库与构建设施。

  * *应对*：采用 Serverless（Supabase/Vercel/CDN）降低门槛，建立标准 SRE 流程。

## 路线图管理原则

* **Schema First**：所有功能扩展必须先行定义 Schema。

* **文档驱动**：每阶段变更以 RFC/文档为准（参考 `00_Project_Management` 目录）。

* **严控边界**：后端逻辑尽量留给 Shopify，Atlas 专注前端体验与内容管理。
