# 项目立项与价值主张 (Business Case)

> **文档定位**：本文档旨在阐述 Jackery Atlas 项目的商业价值、ROI 分析及战略意义，供管理层决策参考。
> **核心目标**：通过自研 Headless CMS 替代昂贵的 AEM/Shopify Theme 体系，实现降本增效与数据资产私有化。

## 1. 为什么现在做？(Why Now)

### 1.1 外部压力：降本需求迫切
*   **去 AEM/CMS 授权费**：传统的企业级 CMS（如 Adobe Experience Manager）年费昂贵。自研 Atlas 可完全替代其内容管理功能，**直接节省数十万/年的软件授权费**。
*   **Shopify 成本优化**：前端完全接管后，不再依赖 Shopify 高级主题模板和部分前端插件（如 Page Builder Apps），未来具备将 Shopify 套餐降级的潜力。

### 1.2 内部驱动：去平台化与资产沉淀
*   **摆脱 Vendor Lock-in**：目前所有页面数据和代码都锁死在 Shopify 生态中。Atlas 将页面结构（JSON Schema）和前端代码（Vue 组件）转化为**公司自有资产**，不再受制于单一平台。
*   **技术栈现代化**：从过时的 Liquid 转向主流的 Vue 3 + Nuxt 3，**提升团队士气**，解决招聘难、留人难的问题（消除 Skill Gap）。

---

## 2. 核心价值主张 (Value Proposition)

### 2.1 降本 (Cost Reduction)
| 成本项 | 现状 (Shopify Native/AEM) | 未来 (Jackery Atlas) | 节省逻辑 |
| :--- | :--- | :--- | :--- |
| **CMS 授权费** | 高昂 (AEM/Contentful) | **零** (自研 + Supabase Free/Pro) | 自研替代采购 |
| **服务器成本** | 包含在 SaaS 费用中 | **极低** (SSG + CDN) | 静态托管几乎无计算成本 |
| **插件月费** | 随安装量激增 (Page Builders) | **大幅降低** | 核心 Builder 功能自研，停用第三方构建插件 |

### 2.2 增效 (Efficiency & Growth)
*   **运营闭环**：非技术人员通过 Visual Builder 自助修改文案/图片，**无需提需求给开发**，缩短营销上线周期。
*   **极致性能**：静态页面 (SSG) 加载速度极快 (Lighthouse 95+)，直接提升 **SEO 排名** 和 **广告转化率 (ROI)**。
*   **数据主权**：前端完全可控，可实现更精细的用户行为埋点 (GA4/CAPI) 和隐私合规 (GDPR)。

## 3. 隐形风险与应对 (Hidden Risks)

在看到收益的同时，必须诚实评估以下隐形投入：

### 3.1 重造轮子成本 (Reinventing the Wheel)
*   **风险**：Shopify 生态中开箱即用的功能（如评论、积分、推荐），在 Headless 模式下前端 UI 全部失效。
*   **应对**：必须预留 **30% 额外工时**用于重写这些功能的前端组件，或寻找提供 API 的替代 App。

### 3.2 运维复杂度 (DevOps)
*   **风险**：不再是“甩手掌柜”，需维护构建流水线、CDN 配置和数据库备份。
*   **应对**：选用成熟的 Serverless 设施 (Vercel/AWS S3/Supabase) 降低运维门槛。

### 3.3 营销灵活性阵痛
*   **风险**：Headless 模式下，修改内容需“保存 -> 构建 -> 发布”，有 5-10 分钟延迟，不如原生 Shopify 即时。
*   **应对**：开发 **增量构建 (ISR)** 能力，并向运营团队做好预期管理。

---

## 4. MVP 策略与退出机制

### 4.1 MVP 聚焦 (Focus)
*   **只做**：单站点首页、产品页、购物车闭环。
*   **不做**：多语言、多币种、复杂营销活动（初期留给 Shopify 原生页）。

### 4.2 退出机制 (Exit Strategy)
如果自研项目失败或效果不达标，如何止损？
*   **方案**：由于 Atlas 是 Headless 架构，Shopify 后端数据始终保持完整。**随时可以将域名切回 Shopify 原生主题**，业务零中断，仅损失前端开发投入。
